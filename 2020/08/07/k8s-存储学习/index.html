7.k8s中的存储学习

Posted on 2020-08-07 Edited on 2020-08-13

k8s中到底有哪些存储机制被我们利用？

ConfigMap描述信息

k8s 1.2版本中引入，因为许多应用程序会从配置文件、命令行参数或者环境变量中读取配置信息 。ConfigMap API 给我们提供了向容器中注入配置信息的机制。

可以保存单个属性，也可以用来保存整个配置文件或者JSON二进制大对象

——>>>>configMap

ConfigMap的创建

  1. 使用目录创建

-from-file 指定在目录下的所有文件都会被用在ConfigMap里的创建的一个键值对，键是文件名，值就是文件内容

也可以使用 kubectl describe cm game-config 查看信息

  1. 使用文件创建

kubectl create configmap game-config-2 --from-file=game.properties --from-file=ui.properties

  1. 使用字面值创建

-from-literal 指定键名键值

  1. 使用yaml创建

vim special-config.yaml

Pod中使用ConfigMap

  1. 使用ConfigMap来注入环境变量

vim env.yaml

vim pod-env.yaml

  1. 使用ConfigMap设置命令行参数
  1. 通过数据卷插件使用ConfigMap

vim pod-test-3.yaml

ConfigMap的热更新

vim config-hot.yaml

kubectl edit cm log-config

Secret

Secret解决了密码、token、密钥等敏感数据的配置问题，而不需要把这些敏感数据暴露到镜像或者Pod Spec中。Secret可以以Volume或者环境变量的方式使用。

如果有些密码需要被pod使用的话，可以先将其存储在secret中，然后在pod中挂载。

Secret有三种类型

  • Service Account：用来访问Kubernetes API，由Kubernetes 自动创建，并且会自动挂载到Pod的 /run/secrets/kubernetes.io/serviceaccount 目录中

    kubernetes API不是随便访问的，pod想要访问需要挂载SA

  • Opaque：base64编码格式的Secret，用来存储密码、密钥等

    加密程度并不高，“表面加密”

  • kubernetes.io/dockerconfigjson: 用来存储私有docker registry 的认证信息

Volume

问题：

容器崩溃时，如果容器被重启此时是以干净的状态重新启动的，也就是之前的文件将会丢失。

其次pod中同时运行多个容器时，需要共享文件 pause

——> Volume

emptyDir

如果是若干个容器一同挂载同一个volume，这几个容器中可以数据共享，在其中一个容器的volume目录下删改，另一个容器中对应的volume目录也会发生变动

hostPath

hostPath 卷将主机节点的文件系统中的文件或目录挂载到集群中

用法：

  • 运行需要访问Docker内部的容器；使用/var/lib/docker的hostPath

  • 在容器中运行cAdvisor；使用/dev/cgroups 的hostPath

  • 运行pod指定给定的hostPath是否应该在pod运行之前存在，是否应被创建，以及他应该以什么形式存在

注意：hostPath Volume为pod挂载宿主机上的目录或文件，使得容器可以使用宿主机的高速文件系统进行存储。缺点是，在k8s中，pod都是动态在各node节点上调度。当一个pod在当前node节点上启动并通过hostPath存储了文件到本地以后，下次调度到另一个节点上启动时，就无法使用在之前节点上存储的文件。
